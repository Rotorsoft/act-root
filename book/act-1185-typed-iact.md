# ACT-1185 — The types that stopped one door short of the user

## The pain that started it

An architectural review traced the framework's type information from the builder DSL to the places application code actually runs, and found that the most carefully engineered typing in the codebase evaporated at exactly the boundary users live in most. The orchestrator-level `Act.do` and `Act.load` are precisely typed through the registry generics: call `app.load(Counter, "c1")` at the top level and the compiler knows `state.count` is a number. Step inside a reaction handler, though, and the `app` you receive is the reaction-scoped `IAct`, whose `do` returned `Promise<Snapshot<any, any>[]>` and whose `load` returned `Snapshot<any, any>`. The machinery to type it correctly already existed one interface away; the interface just never asked for it. Every reaction handler in every application built on Act was reading `any`-typed state, silently exempt from the strictness the rest of the codebase pays for. The same review found two cousins of the problem: the `committed` lifecycle event declared its payload as `Snapshot<TSchemaReg, TEvents>[]`, typing each snapshot's state as the whole action-to-state register map, a shape no snapshot has ever had, which forced the emit site to cast; and the `.given()` hover-docs, the first thing a user sees in their editor when reaching for invariants, showed three-argument function-form predicates returning error strings, an API that has never compiled against the real `Invariant` object shape.

## Why the obvious answer didn't fit

The tempting reshape was to make `IAct` mirror `Act` outright, threading the schema register as a required generic through the reaction pipeline, the builder base, and every handler signature. That would have broken every existing annotation of the form `IAct<Events, Actions>` in user code, turning a type-accuracy fix into a major version event over types that were only ever wrong in the safe direction. The reaction pipeline itself did not need the register either: the win users feel comes from `load`, and `load` carries its own evidence. The `State` argument the caller passes in already knows its schema, so a method-level generic inferred from that argument types the snapshot with no new interface parameter at all. Only `do` genuinely needs the register, because the state a `do` returns belongs to whatever state the target action lives on, unknowable from the action payloads alone. For the `committed` payload, the honest-looking `Snapshot<TSchemaReg[keyof TSchemaReg], TEvents>[]` turned out to be unprovable at the emit site: the compiler cannot show that a key of `TActions` indexes `TSchemaReg` when the register is a free type parameter, so the precise form would have traded one cast for another. And for the doc examples, the seductive path was to conclude the function form was the better API and change `Invariant` itself; that is a real design conversation, but it belongs in an RFC, not smuggled into a documentation repair.

## The decision

`IAct` gained a fourth generic, `TSchemaReg extends SchemaRegister<TActions>`, defaulted to its own constraint so every existing `IAct<E, A>` annotation compiles unchanged, and `do` now returns `Promise<Snapshot<TSchemaReg[TKey], TEvents>[]>` — collapse to the default and that is `Snapshot<Schema, TEvents>`, loose but never lying. `load` was typed with no new generic at all, inferring the snapshot shape from the `State` argument across both the bare-stream and actor-aware overloads, which is where reaction handlers get their casts back. The `committed` payload became `Snapshot<Schema, TEvents>[]`, the widest type that is actually true of every element, and the cast at the emit site simply disappeared; the register parameter stays in `ActLifecycleEvents` underscore-prefixed, holding its slot so no caller's arity breaks. The `Act` class now declares `implements IAct<TEvents, TActions, TActor, TSchemaReg>`, making the orchestrator a faithful superset of the surface reactions see. Every `.given()` example was rewritten to the object form the compiler accepts, with predicates that read only state and the optional actor, because the action payload was never available to invariants and examples that pretend otherwise teach a phantom API. The proof lives in the type-level diagnostics file: a reaction handler loads a state and reads a field with no cast, and a `@ts-expect-error` on a wrong field name must fire or the typecheck fails, which means the `any` collapse cannot quietly return.

## What this teaches

Type engineering is judged at its weakest boundary, not its strongest. A framework can carry registry-level inference through a builder chain with mapped types and conditional inference, and none of it matters to the user whose daily surface is the one interface that says `any`. The repair pattern is worth keeping: prefer generics with defaults over reshapes, prefer inference from arguments already in hand over new parameters, and when the honest precise type is unprovable, choose the widest true type over a cast that asserts the precise one. A loose type is a missed opportunity; a cast is a standing lie waiting for the code to drift underneath it.

## Connections to other chapters

The register generics this threads through were built for the orchestrator in the original builder work, and the discipline of proving type behavior with `@ts-expect-error` diagnostics extends the type-check file that has guarded builder inference since the early slices. The review that found this is the same one that produced ACT-1178 and ACT-1179 — three findings, one habit: read the artifact against the promise, not against the tests that already exist.
