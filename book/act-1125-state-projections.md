# State projections: the list you were always building anyway

Every event-sourced application I have seen ships the same read model first: the list. The orders list, the tickets list, the games list. One row per aggregate, holding the attributes the aggregate already has, so the UI can render a table and a filter box. And every one of those applications builds it the same laborious way: a projection with one handler per event, each handler re-deriving a piece of state the framework already knows how to fold, each handler paying one database write per event. The projection is a worse copy of the reducer sitting a few files away.

Ticket 1125 started from the other end, as a performance ticket. A rebuild replays every event through the projection, so a ten-million-event store pays ten million row writes even when they collapse into fifty thousand distinct rows. The obvious fix is to fold in memory and flush once per key. But the moment you ask what the fold function should be, the two problems reveal themselves as one. The fold you want for the list is the state's own reducer. The framework should not ask users to write it twice.

The surface went through three corrections, each one narrowing it. The first draft introduced a `list()` factory with its own registration verb, and the review killed it in one line: why not keep using projection? A sibling factory splits one idea across two entry points; `projection("orders").of(Order)` says the same thing inside the concept that already exists. The second draft grew a stream filter, because filters feel like generosity, and the review killed that too, with the observation that makes the whole feature coherent: when you project a state you must fold every event of that state, and the scoping you actually want is by state type, which the state itself already expresses. The event register derived from `.of(Order)` is the filter. A fold that skips events of a stream it projects would produce a wrong state, so the option to do so should not exist. The third correction was about fluency: the builder should not offer `.of()` after `.on()` and then throw at runtime; the types should simply not have the method there. A projection either folds a state or declares handlers, and the chain enforces it before the code runs.

The implementation had one insight and two humbling bugs. The insight is that the fold engine compiles into a batch handler. The drain pipeline already knew how to fetch a window of events for a static target, hand them to one function, and acknowledge the watermark only after that function returns. That is exactly the flush-then-ack discipline the fold needs, so the orchestrator did not change at all; the builder emits a long-lived closure and the existing machinery carries it. On first sight of a stream the closure loads its head state through the same `load()` the command path uses, snapshots and cache included, then applies newer events through the reducers as they arrive. During a rebuild the head load makes every replayed event a no-op skip, which is precisely why rebuild cost tracks streams instead of events.

The first bug was the frontier. On a warm cache, `load()` replays nothing and returns no event, and the engine's frontier defaulted to minus one, so every fetched event applied again on top of an already-current state. Counters doubled. The frontier lives in the cache entry, not the snapshot, and the fix was to read it from there. The second bug was the mirror image: replayed events that skipped cleanly also skipped marking the row dirty, so a rebuild folded everything and flushed nothing. An empty table after a perfect replay. The skip now marks dirty, which costs one redundant upsert per stream per round and buys the property that replays always repopulate the table they were asked to repopulate.

The numbers came out the way the write-amplification argument predicted, with one honest wrinkle. On fifty hot streams carrying two thousand events each, the fold rebuilds in under a second with one hundred row writes where the per-event shape takes thirty-one seconds and a hundred thousand. The wrinkle is that a raw `.batch()` handler posts competitive wall-clock, because the benchmark lets it write raw event fields. It cannot produce the folded list at all without hand-rolling the cache, the eviction policy, and the watermark discipline, which is the part of this feature that is not really about speed. It is about owning the subtle correctness code once, in the framework, where the lost-fold bug can be pinned by a test instead of rediscovered in production, one application at a time.
