# ACT-1189 / ACT-1190 — Two ways a Postgres store quietly stops working

## The pain that started it

A second-round adapter scan went looking for the failure modes that never show up in a test suite because they need the outside world to misbehave, and it found two in the Postgres store, both of the same shape: a store that looks perfectly healthy right up until the moment it isn't, and then fails in a way no operator asked for. The first was a connection that drops. The second was a counter that fills up. Neither is exotic — managed Postgres reboots for patching, fails over, and drops idle connections as a matter of routine; a poison stream that never makes progress is exactly what `blockOnError: false` invites. What made both worth a ticket was that the store's own bookkeeping kept insisting everything was fine.

## The crash hiding behind a missing listener

Cross-process wakeups ride a dedicated `LISTEN` client checked out of the pool for the lifetime of the subscription. The subtle part is what node-postgres does at checkout: it removes the idle-error guard it keeps on pooled clients, on the theory that whoever checked the client out will handle its errors. We attached a `notification` listener and nothing else. So when the backend restarted, the client emitted `error`, and an `EventEmitter` with an unhandled `error` does not shrug — it re-raises, and the raise is uncaught, and the process dies. The cruelest version was the near-miss: even a process that somehow survived the throw would hold a client that would never receive another notification, while `_listen_client` still pointed at it looking for all the world like a live subscription. Cross-process reactions would simply stop, silently, with no error and no recovery.

The fix had to do three things that pull against each other. Attach an `error` listener so the throw can't happen. Tear the dead client down and re-`LISTEN` on a fresh one so the subscription heals itself. And make absolutely sure that a reconnect scheduled a quarter-second ago cannot fire after someone has disposed the store — a reconnect that checks out a client against a pool that's being ended is just a different crash.

The wrong turn worth recording is the temptation to reconnect eagerly and synchronously inside the error handler. That trades a crash for a hot loop: a backend that's down stays down for seconds, and a tight reconnect against a refused connection burns CPU and log volume for no benefit. The subscription is a latency optimization over the poll path — the moment it's gone, the poll path is already covering correctness. So the reconnect could afford to be lazy: capped exponential backoff from 250ms to a 30-second ceiling, degrading to polling in between, resetting the backoff only after a `LISTEN` actually succeeds. The self-healing is measured, not frantic, because there is nothing to be frantic about.

Cancellation fell out of the teardown that already existed. `_teardown_listen` clears the caller's handler, and every reconnect path — the error handler, the timer callback, the retry-on-failed-reconnect — bails the instant it finds no handler. Disposal doesn't need to know about reconnects; it just clears the one field they all check. A pending timer gets `clearTimeout`'d for good measure, but the handler guard is what makes the race safe even when the timer has already fired.

## The counter that overflows the whole lane

The second bug lived one query down. `claim()` does `retry = retry + 1` on every acquisition, and the only verbs that reset `retry` are the ones that represent progress — ack, defer, reset, unblock. A stream that fails forever and is configured not to block makes no progress, is re-claimed forever, and marches its retry counter up without bound. On SQLite and InMemory that's harmless — the counter is a 64-bit integer nobody will ever exhaust. On Postgres the column was `smallint`, and at 32,768 the `retry + 1` update raised "smallint out of range." Because the offending row is always claimable, and because `claim()` updates every claimable stream in one statement, that one poisoned row throws the entire multi-stream claim — every subsequent claim in the lane fails until someone does surgery on the row by hand. A silent cross-adapter divergence that turns into a lane-wide outage.

The fix is a one-word type change, `smallint` to `int`, and the whole lesson is in where that word goes. Editing the `CREATE TABLE` is the obvious move and the wrong one: `CREATE TABLE IF NOT EXISTS` never rewrites an existing table, so that change would only ever help databases created after it shipped. Every deployment already in production would keep its `smallint` column and its latent outage. Act has no migration framework and never will — the whole schema-maintenance story is that `seed()` runs on every boot and is additive, idempotent, and lossless. So the widening had to be a ladder step: an `ALTER COLUMN retry TYPE integer` sitting next to the `ADD COLUMN IF NOT EXISTS` migrations, a no-op once the column is already `int`, run by every worker on every boot. The `CREATE TABLE` got the new type too, so fresh databases skip straight to the right shape, but the ladder step is what actually closes the bug for the deployments that have it.

We considered clamping in SQL instead — `LEAST(retry + 1, 32000)` — and rejected it. Clamping papers over the overflow but keeps the divergence: Postgres would cap the counter where the other adapters let it climb, and any operator reading `retry` to reason about a stream's failure history would get a different answer per backend. Widening to `int` doesn't cap anything; it lifts the ceiling to two billion, which for a per-stream retry counter is the same as unbounded, and lands Postgres on exactly the semantics SQLite already had.

## What this teaches

Both bugs are the same bug at different layers: a resource that the store treats as permanent — a checked-out connection, a fixed-width counter — meeting a world that treats nothing as permanent. The connection drops; the counter fills. The store's internal state said "healthy" in both cases right up to the failure, which is why neither surfaced in ordinary testing and why both needed a review that reasons about the environment rather than the happy path. The tests that pin them had to manufacture that environment: a fake pool client that emits `error` on command, and a stream whose `retry` is parked one below the old ceiling by a direct `UPDATE`. Neither condition arises from using the store correctly, which is exactly why both had to be written by hand.

## Connections to other chapters

The self-healing reconnect degrades to the same poll path the oversize-payload guard of ACT-1120 falls back to — `notify` is a latency optimization the whole way down, and every one of its failure modes is a latency regression, never a correctness one. The seed-ladder discipline the retry widening rides is ACT-1140's "seed is the schema story" made concrete one more time: the interesting schema changes aren't the new columns, they're the type changes that have to reach the databases that already exist.
