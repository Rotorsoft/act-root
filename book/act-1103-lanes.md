# ACT-1103 — drain lanes

ACT-1103 ships drain lanes: `.withLane({...})` on the builder, `.to({lane})` on reactions, one `DrainController` per declared lane, parallel `_drainAll` fan-out, per-lane `setTimeout` workers, an optional `lane` filter on `Store.claim`, and a `streams.lane` column on every adapter. The headline number on Postgres is ~7× faster fast-event latency under slow-lane backpressure. The essay sits in the scaling chapter, after dual-frontier (ACT-103) and priority lanes (ACT-102), as the answer to "what do you do when reactions have heterogeneous timing profiles?"

---

### Threads to develop

**1. The pre-1103 shape and what it cost.**
Before lanes, Act ran one drain pipeline with one global `leaseMillis`. That works fine when reactions have similar timing — every handler completes in tens of milliseconds, the lease window is comfortable for all of them, retries pace evenly. It breaks the moment they don't. A webhook reaction wanting a 30-second lease for slow external receivers forces the same 30-second window on every other reaction in the system; a snappy projection that should retry in a second waits half a minute. The pragmatic answer until ACT-1103 was "tune `leaseMillis` to the slowest handler and live with the slow recovery for the fast ones." The book should make the cost concrete: a fast reaction that fails transiently waits the full `leaseMillis` before the next attempt, because backoff's floor is `max(configured, leaseMillis)` — the controller holds the lease during the window. Bumping the global lease to fit one slow handler degrades every other reaction's retry latency.

**2. The "obvious" design that wasn't right.**
The first sketch was a multi-lane `claim()` — one SQL call returns leases from N lanes via an N-way `UNION`, the orchestrator arbitrates allocation ratios per cycle, all the lane bookkeeping lives in the store adapter. It's seductive because it sounds like a clean primitive: "make the existing claim multi-lane aware." The cost: every adapter changes shape (a breaking port change), the SQL grows considerably, and the framework has to express "how much of each lane?" as a single function signature. After sketching, the picture was uglier than it looked from the headline — the SQL planner would have to coordinate per-lane budgets in one statement, the InMemory adapter would have to mirror that logic, and operators would tune ratios as a single black-box parameter. The book should walk through this rejected design honestly because it's where most readers will arrive first.

**3. The cheaper architecture.**
The shift was reframing the unit of work: instead of one drain controller that handles many lanes, one drain controller *is* a lane. The controller already owned `_armed` / `_locked` / `_ratio` / backoff state per process; adding lanes just means instantiating N of them, each filtering its claim by a single lane name. The Store contract gains exactly one optional parameter (`Store.claim(..., lane?: string)`), the adapter SQL collapses to the pre-1103 shape when `lane` is undefined, and every existing optimization (`_armed` skip, dual-frontier split, per-stream backoff) automatically applies per lane without modification. The point worth lingering on for the book: the right architecture often comes from asking "what's already in place that I can just have more of?" rather than "what new abstraction do I introduce?" The DrainController was a complete unit; the lane work is *replicating* it, not *augmenting* it.

**4. Lane is a property of the stream, not the reaction.**
This is the load-bearing design call. The alternative — lane on the reaction — would force `claim()` to know about reaction definitions to decide which lane to filter by. By putting lane on the stream (one row in `streams.lane`), `claim(lane=X)` becomes a flat `WHERE lane = X` predicate served from an index. Every operation that operates on a set of streams (`reset`, `unblock`, `prioritize`, `query_streams`) takes the same `lane` filter on its `StreamFilter` for free. The downside: two reactions targeting the same stream must agree on the lane. Disagreement is a config error and throws at `act().build()`. The book should explain why this is the *correct* constraint and not a workaround — lanes have no ordering, so there's no analogue of priority's `max()` merge. A stream is in a lane or it isn't; if two reactions disagree, one of them is wrong about its target.

**5. Re-laning on restart — the spec reversal.**
The original ticket said "lane is set at first subscribe and immutable thereafter." The reasoning was simplicity: once a stream picks a lane, no migration is ever needed, the SQL stays trivial. In practice that's the wrong default. Operators want to move a target to a different lane by editing the config and restarting; the alternative ("drop the stream, change the lane, replay") is a footgun nobody should reach for. The reversed spec: `subscribe()` UPSERTs the row's lane on every call. At restart, the new builder lane wins; the old worker is dead so there's no in-flight lease conflict. The book should explain why this works without paying the migration cost we feared: the cost was only the *online* re-laning case (changing the lane while workers hold leases), which we still don't support. The cheap case — restart-driven — covers the operator's real need. The lesson generalizes: when an "immutable" spec line is load-bearing for simplicity but harmful in practice, look for an alternate trigger (here: process restart) that resolves it without paying the original cost.

**6. The slice-typing gap.**
Slice 1 shipped the builder fanout with strong typing on inline reactions: `.to({lane: "slow"})` narrowed to declared lanes, typos failed compile. But slices couldn't see the parent Act's lane set, so their `.to({lane})` accepted any string and relied on a runtime gate at `act().build()`. The user caught this gap in review: "should at least catch type errors by forcing withLane in slices." The fix landed as slice 1b — `SliceBuilder` carries its own `TLanes` generic, slices declare their own lanes via the same `.withLane(...)` method, and `act().withSlice(slice)` merges the slice's lanes into the Act's set with conflict detection on timing. The runtime gate stayed as a backstop for slices built against older type definitions (npm-installed older versions). The book should use this as an example of how soundness gaps in fluent APIs hide behind "we'll validate at runtime" — and how closing them at the type level changes the maintenance shape: instead of an ongoing runtime cost, the constraint becomes invisible once the types compose right.

**7. Sequential vs parallel drain — the late discovery.**
Slice 7 wired one controller per lane and changed `Act.drain()` to fan out across them via a sequential `for await` loop. The fan-out tests passed; the overhead bench showed a tolerable ~7% on InMemory. But the responsiveness bench told a different story: in early runs, two-lane and single-controller configurations produced identical fast-event latency (~130 ms — the slow-handler floor). The reason was sequential drain: `_drainAll`'s `for await` made the slow lane's 100 ms-long drain block the fast lane's drain in the same `Act.drain()` call. Promise-all across controllers fixed it in a one-line change, and the bench then showed the ~7× speedup the architecture had implied all along. The book worth flagging here: parallel-across-controllers is *safe* because each lane filters its own `claim()`, and the store's `SKIP LOCKED` keeps cross-controller races correct. Sequential was the wrong default by accident — `for await` is the easy-to-write pattern that hides the latency penalty. The lesson: when the design says "these operate independently," the implementation has to actually let them.

**8. Per-lane workers.**
The earlier shape relied on the Act's single `SettleLoop` to drive every controller. That's enough when callers explicitly `await app.settle()` after commits, but it means a lane's responsiveness still depends on whatever cadence settle runs at. The user asked: "should each lane have its own async worker with interval of lane?" The answer is yes — `LaneConfig.cycleMs`, when set, auto-starts a `setTimeout` chain on the controller that calls its own `drain()` at the configured cadence. The timer is `unref()`'d so it doesn't keep the process alive on its own, and `Act.shutdown()` clears it. The teachable framing: the Act-level settle loop is for *commits that callers can correlate with reactions*; the per-lane worker is for *lanes that should drain continuously regardless of caller activity*. They compose — both can be active at once — but most apps will end up using one or the other per lane.

**9. The LaneConfig-vs-options precedence question.**
A subtle question surfaced during slice 7b: when `withLane({leaseMillis: 30_000})` is declared and a caller passes `drain({leaseMillis: 1_000})`, which wins? The intuitive answer (caller options win) makes lanes pointless — the whole reason `withLane` exists is to give the lane its own budget. The corrected semantic: lane config wins; caller options apply only where the lane is silent. This is the kind of small precedence call that quietly determines whether a feature *works* in production. The book should make it explicit: the right default is the one that respects the more specific declaration, even when that flips the usual "caller wins" intuition.

**10. The InMemory vs Postgres benchmark surprise.**
The first overhead bench ran against `InMemoryStore` because it was the closest to hand. Numbers: +7% mean for 4-lane vs 1-lane drain. That looked reasonable as "the cost of fan-out." The user pushed back: benchmarks belong on real adapters. The PG bench told a different story: +1% to +4% across two runs, comfortably inside run-to-run variance. The reason is the lane filter: on PG, `claim(lane=X)` serves from `streams_lane_ix` (a plain btree on the `lane` column); four parallel claims add up to the same total work the single all-lanes claim was doing. On InMemory, every `claim()` scans every subscribed stream and rejects those outside its lane — a linear filter with no index. InMemory is the pessimistic case for the fan-out filter; PG is the production-shape number. The book should make this contrast a teaching point: InMemory makes a fine semantic adapter for tests but a bad oracle for performance work, because its naive O(n) loops mask the wins that real adapters get from indexes. CLAUDE.md gained a rule out of this: every perf claim that ships in a PERFORMANCE.md table needs numbers from `act-pg` or `act-sqlite`; InMemory may appear as a baseline reference, never as the primary.

**11. Lifecycle events carry lane — listeners filter.**
The orchestrator's `acked` / `blocked` / `settled` events fire per controller, each carrying leases that include `lane`. Listeners that care about a specific lane filter on `lease.lane`. This kept the lifecycle emit semantics unchanged — the alternative (one event per lane, named like `acked:slow`) would have multiplied the public surface and forced every listener to subscribe to N events. The book worth highlighting: when adding a dimension (lane) to an event, the right move is usually to thread it through the payload, not to spawn new event channels. New channels are tempting because they make filtering cheap; they make the surface expensive.

**12. `onlyLanes` as escape hatch, not primary path.**
The ticket originally framed lanes around three deployment shapes: single process running all lanes, process per lane, and arbitrary partitioning via `ACT_ONLY_LANES`. After slice 7b landed parallel `_drainAll` and per-lane workers, the responsiveness story made the single-process case the primary path — fast-lane events ack in ~17 ms while slow-lane handlers churn for 100 ms+, all in one process. `ACT_ONLY_LANES` (and its programmatic equivalent `ActOptions.onlyLanes`) became the escape hatch for hardware-isolation needs (different CPU/memory per lane), not the default deployment recipe. The book should explain how this shift happened: a feature shipped with three deployment shapes in mind ends up with one primary and two specialized. That's a normal outcome of running benchmarks early — you find out which shapes actually pay off.

**13. The TCK consolidation lesson.**
The first three adapter slices (InMemoryStore, PG, SQLite) each got their own per-adapter `lanes.spec.ts` covering subscribe defaults, claim filter, query_streams projection, prioritize/reset/unblock filtering. Three sets of nearly-identical tests. The user pointed out the redundancy — and that the TCK in `libs/act-tck/src/store-tck.ts` is the executable contract every adapter validates against. CLAUDE.md spells out the "Changing a port interface" rule explicitly: when a Store / Cache / Logger method changes, the TCK changes in the same slice. The fix retro-collapsed the three per-adapter test files into one TCK block, leaving per-adapter spec files for things only that adapter cares about (PG's idempotent `ALTER TABLE` migration, SQLite's same). The lesson worth keeping in the book: it's easier to write the per-adapter test first because you're working in that file already; it's wrong because the contract belongs in the contract module. CLAUDE.md gained a more visible "Before you start" section out of this — port-interface changes ride with the TCK update, full stop.

**14. The headline number, restated.**
Two single-Act configurations against the same Postgres store: one with no `withLane(...)`, one with `slow` + `fast` lanes. Workload: 5 events on slow streams (handler sleeps 100 ms) + 50 events on fast streams (no-op handler). Measured: commit-to-handler-fires latency for the fast events. Single controller: 130–133 ms p50, 134–140 ms p95. Two lanes: 17.0–18.6 ms p50, 17.9–20.9 ms p95. ~7× speedup. The shape of the win is structural: the single controller's `runDrainCycle` does `Promise.all` across handlers within a cycle, which means all leases ack at the same time — and that time is the slowest handler's. The two-lane setup runs each controller's `runDrainCycle` independently, so fast handlers ack on their own short cycle while slow handlers churn separately. The book should land this number prominently. It's the answer to "why are lanes worth the complexity?"
