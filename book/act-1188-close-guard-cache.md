# ACT-1188 — The cache that remembered a stream was closed and then forgot

## The pain that started it

A second-round core scan went reading the load and commit paths against the close-the-books guarantee, and found a window where the two disagreed. Closing a stream happens in two moves. First a tombstone is committed so every later action sees a closed head and bounces off it. Then, once the archive callback has drained the old events somewhere safe, a truncate deletes them. Between those two moves there is a guard window, and it can stay open for as long as the archive takes, or forever if the close is aborted and the tombstone is left standing on purpose.

Inside that window a cold load is the villain. When nothing is cached, load replays the stream from the beginning, folds the real events, walks over the tombstone at the head, and then does the thing it always does after processing events: it writes a checkpoint into the cache. That checkpoint records the state, the version, and the event id of the head, which is the tombstone. What it does not record is the head event itself, because the cache entry has no room for it. So the next action loaded the stream, got a warm hit, and asked the one question that gates a closed stream, whether the head event is a tombstone. The head event on a warm hit is undefined. The question answered no. The action committed a new event past the tombstone, at the tombstone's version, and no adapter refuses an append past a tombstone because the tombstone check was supposed to live in the orchestrator. Then the truncate arrived and deleted everything below the boundary, including the commit that had just been acknowledged to a caller. Silent data loss, dressed up as a successful write.

## Why the obvious answer didn't fit

The instinct is to teach the cache to carry the head event so the warm-hit check has something to read. That is more surface for a checkpoint that exists precisely to be small, and it invites the same class of bug from a different direction, because now every warm hit has to be trusted to reconstruct the closed state correctly rather than simply not existing. The cache was never meant to be the source of truth about whether a stream is closed. The store is. The warm-hit path was only ever safe because, in every ordinary case, a warm entry means the head is a real domain event and the tombstone question is asked against a cold load that still has the raw head in hand.

The real defect is that the guard-window load wrote a cache entry at all. In the already-safe cases it never does. A truncated stream replays only the tombstone, which folds to nothing, so the replayed count stays zero and the write is skipped. A pre-warmed stream also replays nothing past its checkpoint and skips. Only the cold-load-during-guard case replays real events, bumps the count above zero, and then poisons the cache on the way out. The fix is to notice when the replayed head is the tombstone and decline to cache. Leaving the entry cold is not a workaround. It is the correct state, because a cold entry forces the next action back onto the load path that still holds the raw tombstone and asks the question against something real. The check stays live, the action throws, and the acknowledged-then-deleted commit can never happen.

## Two more from the same reading

The scan turned up two neighbors while it was in these files. Scoped Acts, the ones that carry their own store and cache through an async-local bag so a single process can serve many tenants, leaked to the singleton on three background paths. Settle drove its correlate init without the scope wrapper, so the first cold start subscribed a tenant's static targets on the shared singleton and then marked itself initialized, which meant the scoped store never learned about its own reaction targets. The correlation poll timer and the per-lane worker ticks had the same shape, firing outside any caller frame and resolving the ports to the singleton. The pattern that hid all three is that the foreground methods already wrapped their work in the scope, so the ordinary tests, which drive foreground calls, saw nothing wrong. Only a static target watched on the wrong store exposes it. The fix threads the same scope runner the foreground uses into the two internal cycles and the settle init, and makes that runner a required constructor argument rather than an optional one, so there is no identity default for a future caller to forget to override.

The third neighbor lived in restore. When a restore renumbers event ids, because dropping snapshots or any other drop shifts the id space, the scan remaps each event's causation pointer to the new id. That remap rebuilt the event from the original row instead of from the migrated one, so any event that both carried a schema migration and pointed at an earlier shifted event silently reverted to its pre-migration name and data. The migration had run a few lines above and was then thrown away. Spreading the migrated event instead of the original is the whole fix, and it is the kind of one-word correction that only a test composing a drop with a migration would ever have caught.

## What this teaches

Each of these is a disagreement between two representations of the same fact. Whether a stream is closed is a fact the store owns and the cache was caching a stale shadow of. Which ports a scoped Act uses is a fact the scope bag owns and three timers were reading from the wrong place. What an event's name and data are after a migration is a fact the migrated record owns and the remap was rebuilding from the record it had already superseded. The bugs were invisible at every altitude except the one where a background path, a guard window, or an id shift makes the two representations drift apart. Reading code against the guarantee it claims to uphold, rather than against the green tests that already pass, is how you get to that altitude, and it is why the behavior-contract ledger now carries a row for the close-guard load and a row for the scoped background paths, each pinned to the tests that would fail the moment the two representations disagree again.
