# ACT-1222 — The archive that ran twice because two closers didn't know about each other

## The pain that started it

A wave-2 concurrency scan went looking for places where two writers could touch the same stream without one excluding the other, and it found one hiding inside the lightest, most carefully-justified path in the close machinery. Windowed close is the prune that keeps a long-lived stream inside a rolling retention window: it deletes the prefix of events older than a cutoff, leaves the boundary snapshot standing, and lets the stream keep committing at the head. The whole design leans on one observation, that the cutoff is always in the past, so a concurrently-written snapshot carries a creation time of now and can never become the boundary. Once the cutoff is fixed the boundary is fixed and the prefix below it is immutable. From that, the branch earns the right to skip the tombstone guard that the full close path pays for. No guard commit, no seed, no cache touch. A strictly lighter close.

That reasoning is sound, and it is also incomplete, because it only defends against appends. It quietly assumes there is a single closer per stream. In production there are two. An autoclose rolling-window policy fires its prune from a reaction that runs under a synthetic `__autoclose__:X` drain lease, so no two autoclose passes for the same aggregate overlap. But an operator who calls `app.close([{ stream: X, before }])` for a one-off prune runs `run_close_cycle` directly, on the same Act instance, and never takes that lease. The lease was the only thing keeping the two apart, and the manual path walks right past it. Both closers enter the windowed branch for the same stream at the same time, and the branch, being guard-free by design, lets them both proceed. The prefix delete is idempotent, so the store survives the double truncate none the worse. The user's archive callback is not. It runs twice against the same prefix, which for a real archiver means a double upload to S3 or a duplicated append to a JSONL cold-tier file. The framework told the host to persist the same range twice and the host obliged.

## Why the guard wasn't the answer

The reflex is to give the windowed branch the tombstone guard the full path has, so the second closer bounces off a closed head. That is exactly the wrong direction. The guard exists to freeze a stream that is about to vanish, and a windowed close is defined by the stream not vanishing. Reintroducing the guard would tombstone a stream that must stay live, break the prune-not-retire contract that the whole feature is built to honor, and pay the guard-commit cost that the branch was designed to avoid. The append immutability argument is still correct. The gap is not that appends can race the prune. It is that two prunes can race each other, and specifically that each one independently decides to run the archive.

So the fix has to coordinate closers without touching the store's notion of a closed stream. Two facts make that tractable. The first is that both racers run on the same Act instance, in the same process, because a manual close and a drain's autoclose both go through the one orchestrator that owns the cycle. A process-local lock is enough. There is no cross-process closer to worry about, because the only cross-process closers are autoclose reactions and those already exclude each other through the drain lease. The second fact is that a windowed prune, once it has run, leaves a fingerprint the next closer can read. After the prefix below the boundary is deleted, the boundary snapshot becomes the earliest surviving event. A closer that arrives afterward, looks for the same boundary, and asks whether anything sorts below it, finds nothing. The prune it would perform is empty.

## The shape of the fix

The orchestrator threads a per-stream lock into the close cycle, shared across `app.close` and the drain's `on_close` because they are methods on the same instance. Windowed work for a given stream chains behind whatever windowed work is already in flight for that stream, while different streams still run in parallel. That serialization on its own does not remove the double archive, because the second closer, once it acquires the lock, would still run its archive unless something told it not to. The something is a read-only prune-pending probe that runs inside the critical section, resolves the boundary the truncate would pick, and reports whether any event sorts below it. If the answer is no, either because no snapshot qualifies or because the prefix has already been pruned, the closer skips its archive entirely and reports the stream in the same `skipped` bucket a no-op truncate always used. The first closer through sees a real prefix, archives it, and truncates. The second, serialized behind it, sees the boundary sitting at the earliest event, recognizes there is nothing left to prune, and declines. Archive at most once per pruned range, and the days-only, prune-not-retire semantics are untouched.

The probe deliberately mirrors the store's own boundary rule rather than inventing a new one, so the orchestrator's idea of whether a prune is pending never drifts from what the truncate will actually do. That is a small duplication of the store's snapshot-selection logic, pinned by the existing windowed tests that exercise the same boundary from the truncate side, and it buys the archive gate the ability to run before the events are gone, which the archiver requires because its whole job is to read the prefix it is about to lose.

## What this teaches

The bug was not in the immutability argument, which is correct, but in the scope of the invariant it was quietly relying on. Guard-free was justified against the hazard the author was thinking about, concurrent appends, and silent against the hazard they were not, concurrent closers. A lease was doing the load-bearing exclusion, and the moment a second entry point sidestepped the lease the guarantee evaporated with nothing executable to notice. The behavior-contract ledger now carries a row that fails the instant two windowed closers fire the archive twice for the same range, so the single-closer assumption is written down as a checked fact rather than an unstated premise. When a path earns the right to skip a guard, the thing worth writing down is not only why the guard is unnecessary but exactly which other writers the missing guard is trusting someone else to keep away.
