# The native compiler arrives, and the DSL doesn't flinch

## The pain that started it

A type-heavy framework pays a tax on every keystroke. Act leans on TypeScript harder than most, because the builder DSL is the product. When someone writes `.emits({ TicketOpened })` and then `.patch({ TicketOpened: ... })`, the payload type flows from the first call to the second through a chain of conditional and mapped types that the language server has to re-evaluate on every edit. Whole-workspace typechecks are the slow gate in the merge pipeline, and the editor's own inference lag is the tax the maintainers pay all day. TypeScript 7 is the Go rewrite of the compiler, the same `tsc` binary but native, and its whole reason to exist is to make that tax smaller. The question was never whether we wanted it. The question was whether a codebase this type-forward could move without a week of type errors.

## Why the obvious worry didn't materialize

The obvious worry with any compiler rewrite is that a new type checker means a new definition of "well typed," and that the corners a framework like this lives in are exactly the corners where two implementations diverge. Conditional type distribution, variance in the deep generic signatures of the state and slice builders, the inference that makes `reactingTo` land as the right event shape. If TS7 disagreed with TS6 about any of it, the diff would be a scattering of red across the parts of the code that are hardest to reason about and least safe to loosen.

It didn't happen. A spike ran the whole-workspace typecheck on the TS7 GA build and came back with zero errors, and the full migration confirmed it. Nothing in `libs/act/src` needed a cast, a widened type, or an `any` to satisfy the new checker. The declaration-emit path that produces the published `.d.ts` files, the part most likely to expose a divergence because it forces the checker to name every inferred type out loud, emitted cleanly for every library and not just the trivial ones. The reason is worth stating plainly, because it is the whole argument for writing types this way. A DSL built on the language's own inference rules, rather than on clever tricks that happen to work in one compiler version, is portable across implementations of those rules by construction. The type-forward design paid for itself the day a second compiler had to agree with the first.

## The one place we had to compromise

The framework moved in one commit's worth of `package.json` edits. The docs did not, and that is the only interesting decision in the whole migration. The API reference is generated by typedoc, which reads the framework's types and renders them to markdown, and typedoc's latest release caps its `typescript` peer dependency at the 6.x line. It cannot load the TS7 compiler API. So the choice was to hold the entire workspace back on TS6 until typedoc catches up, or to let the one package that depends on typedoc keep its own TS6 while everything else moves.

We took the island. The `docs` package alone declares `typescript@6`, and pnpm's per-package isolation does the rest without a single override or hoisting exclusion. The docs directory gets its own symlink to a TS6 install while the root of the workspace resolves TS7, and a one-line check after install confirms each side sees the version it should. The snippet type-checker, which compiles every fenced code block in the docs against the live framework source, runs on that same TS6 island and passes, because the snippets exercise the public API surface rather than any compiler-version corner. The compromise is real but small and, more importantly, self-liquidating. It is one dependency on one package, documented as temporary, waiting on an upstream release that is already in motion. When typedoc supports TS7 the island dissolves into a one-line bump.

## What this teaches

Two lessons sit on top of each other here. The first is that the payoff of writing types against the language's own rules is not felt until the rules get a second implementation, and then it is felt all at once as the absence of work. The second is that a toolchain migration is rarely blocked by the thing you are migrating and usually blocked by something downstream of it. The compiler was ready. The one package that reads the compiler's output was not, and the entire shape of the migration is the shape of routing around that single lagging dependency without holding the rest of the codebase hostage to it.

## Connections to other chapters

The stability snapshots that would have screamed if the public surface shifted are source-text based, which is why a compiler swap that changes how types are emitted leaves them silent. That machinery is the same `runStabilityTck` guard the naming-convention and charter work leans on throughout, doing here what it was built for from the other direction, confirming a change that touches everything changed nothing that readers can see.
