# ACT-1220 — The over-correction that banned patterns and broke the demo

## The pain that started it

A prior fix went looking for an overmatch and found a real one. The in-memory store compiled a subscription's `source` into an unanchored regular expression, so a literal-looking source `s1` quietly matched a sibling stream `s12` that shared its prefix. That was a genuine bug, and the fix was correct to want an exact match there. But the fix reached for the wrong lever. It declared that `source` in the claim path was exact-only, always, and rewrote every adapter to compare it by string equality alone. The premise underneath was that every shipped resolver produces exact stream names. It does not. The calculator example — the framework's own front-door demo — registers a static reaction whose source is `^(A|B)$`, a real anchored regex, and its README documents source-regex resolvers as a feature. Once claim compared sources by equality, the has-work probe asked its map for `^(A|B)$`, got nothing, and the Board projection stopped being claimed. The demo silently stopped projecting, and CI stayed green because no test exercised the Board.

## Why the obvious answer didn't fit

The tempting response was to pick a side. Go back to regex everywhere and the overmatch returns: `s1` claims `s12` again, and the fix that started all this is undone. Stay exact-only and the shipped, documented regex feature stays broken. Both answers treat the source as one kind of thing, when it was always two. A bare stream name and an anchored alternation are not the same object wearing different clothes; they are a literal and a pattern, and the store had been guessing which was which by running everything through the same matcher. The overmatch was a literal being treated as a pattern. The broken demo was a pattern being treated as a literal. One matcher could never be right for both.

## The decision

The classification became explicit. A single helper decides whether a source carries any regex metacharacter, and that one bit routes every matching site. A literal source takes the fast, index-friendly equality path — exact, so `s1` never touches `s12`, and cheap, so the perf win that motivated the original fix survives. A pattern source is compiled and matched against candidate stream names with the caller's own anchoring, so `^(A|B)$` claims A and B and nothing else. The same bit threads through the drain's fetch, which had been reading events with an unanchored regex regardless: a literal source now fetches with `stream_exact`, closing the last gap where an exact source could still pull a sibling's events into a handler. Postgres carries the split into SQL, keeping equality on the index for literals and the POSIX operator for patterns. SQLite is the honest exception. Its engine has no regex and its portable LIKE grammar cannot express alternation, so rather than silently never claiming a stream it cannot match, it throws at subscribe time, at registration, pointing the operator to the stores that can. The proof that the bug was real lives in two red-first tests that fail on the old code and pass on the new: the store-level pattern-source claim, and an end-to-end assertion that the calculator's Board projection counts digits from both streams.

## What this teaches

An overmatch and a missing match look like opposite bugs, but here they were the same bug seen from two sides — a matcher asked to serve two kinds of input with one rule. The exact-only decision fixed the half it could see and broke the half it could not, and the half it could not see was the shipped demo, invisible because nothing tested it. The lesson is to name the distinction the data already contains before choosing a matcher, and to own the correction plainly: the earlier fix was right about the overmatch and wrong to conclude that patterns had no place in claim. Literal detection is the contract that satisfies exact-safety, regex support, and the index all at once, because it stops pretending the two were ever one.
