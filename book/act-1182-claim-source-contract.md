# ACT-1182 — Three stores, three opinions about one string

## The pain that started it

An architectural review asked a question nobody had asked in two years: when a subscription carries a `source`, what does it actually mean inside `claim()`? The answer depended on which adapter you asked. Postgres compared it to event stream names with plain equality. The in-memory store compiled it into a JavaScript regular expression, unanchored, and scanned every stream for a match. SQLite translated it through the portable LIKE grammar, which wraps a bare literal in `%` wildcards. Three implementations of one port method, three different contracts, and the divergence had a bite on both ends. A pattern-shaped source like `^order-.*` worked on the two embedded stores and silently starved on Postgres after its first ack, because the exact-equality probe never matched again and the stream never had work. Worse, the unanchored matching meant an innocent exact-looking source suffered substring overmatch: a subscription sourced from stream `s1` woke up whenever `s12` committed, on the in-memory store and on SQLite alike. Nothing pinned either semantic, because the TCK's claim suite only ever exercised exact names that happened to behave the same everywhere.

## Why the obvious answer didn't fit

The seductive fix was to bless the richer semantic and teach Postgres the portable filter grammar, the same one `query_streams`, `reset`, and `unblock` already speak. It reads like unification, and it is the direction a reviewer instinctively leans, because patterns are strictly more expressive than names. But expressiveness nobody uses is pure liability. Every resolver the framework ships, from autoclose to dynamic per-aggregate reactions, returns an exact stream name as its source; no code path anywhere produces a pattern. Meanwhile the claim probe is the hottest query in the reaction pipeline, and Postgres keeps it fast precisely because `e.stream = s.source` rides the stream index. Swapping equality for `~` would trade a measured hot path for a hypothetical feature, and the substring overmatch would remain legal behavior on the stores that regex-matched, just waiting for a tenant named `user-1` to receive `user-12`'s wakeups. The other tempting move was to leave the code alone and only fix the documentation, but a contract that lives in prose while three adapters disagree is exactly how this hole appeared in the first place.

## The decision

`source` is exact-only in the claim path, everywhere. The in-memory store's regex cache is gone, replaced by a single map lookup against its per-stream max-id index, which is both correct and faster. SQLite's probe now compares with `=` instead of a contains-LIKE. Postgres is untouched, because it had been right all along. The pattern grammar survives where it belongs and where it is genuinely used: the StreamFilter surfaces that drive operator queries and bulk recovery. The close cycle's safety probe keeps regex-matching stored sources in process, which is sound for exact names and errs conservative when it errs at all. The TCK grew a claim-source suite that fails red on the old in-memory and SQLite behavior, the prefix-sibling case verified failing before the fix landed, and the behavior-contracts ledger now carries the row so the semantic cannot drift apart a second time. The same review swept up two neighbors: the lease-timeout retry accounting where the docs claimed the opposite of what every adapter did, resolved in the code's favor because repeated worker deaths on one stream are poison-adjacent and should march toward the block, and the notify self-filtering MUST that the port stated and the TCK never checked, now enforced with two sibling instances against one backing store.

## What this teaches

A port contract is only as specified as the inputs its conformance suite feeds it. All three adapters passed the same TCK for years while disagreeing about `source`, because the suite's test data lived inside the intersection of their behaviors. The fix for that class of bug is not more careful reading, it is adversarial test data, inputs chosen to land in the difference between implementations rather than the overlap. And when a contract dispute surfaces, the winner is not the most expressive interpretation but the one the actual callers rely on: exactness was what every resolver meant, what the index wanted, and what the overmatch victims assumed.
